# Community participation

Aurel's public GitHub participation should make the surrounding ecosystem better, not merely make the account look busy.

## Useful participation

- reproduce a real bug against the current default branch;
- report the smallest useful test case and environment facts;
- repair documentation where the mismatch is verified;
- submit focused code changes with tests;
- review a pull request only when there is enough evidence to add value;
- maintain a fork only while a contribution or retained patch is active;
- publish a small evaluation or tool when it is genuinely reusable.

## Avoid

- filler commits for contribution-graph volume;
- generic praise or promotional comments;
- AI-generated issue floods;
- cross-posting the same idea without project-specific evidence;
- presenting a fork, bookmark, experiment, or private result as adopted production capability;
- asking maintainers to debug an unreproduced local assumption.

## Sustainable cadence

The default target is one meaningful public contribution or maintained artifact per week. Quality and relevance outrank frequency.

## Collaboration proposals

A useful issue includes:

1. the real problem or opportunity;
2. direct source links;
3. relevant constraints;
4. the desired outcome;
5. what can be verified publicly;
6. any privacy or authority boundary.

Never include credentials, customer-private material, recovery data, or wallet secrets.
